# Phase2-05/06/07 Simulator, Revision Restore, and Test Report

## Scope

This overlay closes the remaining Phase 2 tail work that can be completed without real voice, WebSocket, OTA activation, or wake word runtime.

It covers:

- Phase2-05 local tool simulator and command log viewer polish.
- Phase2-06 revision validation and minimal restore entry.
- Phase2-07 risk policy test coverage and completion report.

## Implemented

### Phase2-05

- The settings debug panel remains a local tool simulator, not a normal note editor.
- It supports manual `tool name` and `arguments JSON` entry.
- It shows command status, risk level, confirmation status, message, command log id, confirmation id, affected note ids, affected tag ids, error code, and result JSON.
- It keeps confirm/reject controls for pending high-risk commands.
- The recent command log viewer now displays source, tool name, status, risk, confirmation status, timestamps, error message, affected notes, affected tags, and result preview.

### Phase2-06

- Added a revision debug box in Settings.
- Users can enter `note_id` and refresh recent revisions.
- The panel shows revision id, reason, source, command log id, title snapshot, content snapshot, tags snapshot, and state snapshot.
- Users can apply a `notes.restore_revision` simulator sample from the selected note/revision.
- `notes.restore_revision` is implemented as a high-risk command.
- Restore revision returns `requires_confirmation` before mutation.
- Confirmed restore snapshots the current state before restoring the target revision.
- Restore validates note state through `expected_updated_at` to avoid overwriting a note changed after preview.

### Phase2-07

- Added unit tests for `NoteRiskPolicy` covering low/medium/high risk tools, tag bind modes, batch risk escalation, and tag rename escalation.
- Added `testImplementation(libs.junit)` to `notes-domain`.

## Manual acceptance

1. Create a note with the simulator.
2. Append content or update title to create at least one revision.
3. Enter the created `note_id` in the Revision panel and refresh.
4. Confirm the panel lists revision entries.
5. Use the restore sample generated by the panel.
6. Execute it and verify it returns `requires_confirmation`.
7. Confirm it and verify the note is restored.
8. Verify the recent command log shows `notes.restore_revision` with high risk and confirmed status.
