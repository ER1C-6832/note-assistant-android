# Phase1-04 搜索与筛选交付报告

## 1. 本阶段目标

让便签真正好找，在现有 Phase1-03 的标签、待办、置顶和最近删除能力上，补齐标题 / 正文 / 标签搜索，以及搜索和筛选组合后的列表体验。

## 2. 本阶段完成内容

- 将首页搜索框从静态占位改成真实输入框。
- 搜索输入每次新增、删除或修改字符都会立即刷新列表结果。
- 支持搜索标题、正文和标签。
- 支持搜索与筛选组合：
  - 全部 + 搜索
  - 待办 + 搜索
  - 已完成 + 搜索
  - 置顶 + 搜索
  - 单个 tag + 搜索
  - 最近删除 + 搜索
- 增加搜索结果提示条，显示当前范围、关键词和结果数量。
- 增加搜索清除入口。
- 搜索空结果时显示明确的空状态，而不是空白列表。
- 保留 tag 筛选下新建便签自动带入当前 tag 的能力。

## 6. UI 变更

- 搜索框可输入、可清除。
- 输入关键词后会显示当前范围和结果数量。
- 没有搜索结果时显示“没有找到相关便签”。
- 保持左侧 tag 抽屉、顶部筛选 chip、新建按钮和语音占位按钮不变。

## 7. 搜索排序规则

搜索命中后按以下顺序排序：

1. 标题完全匹配。
2. 标题包含关键词。
3. 标签完全匹配。
4. 标签包含关键词。
5. 正文包含关键词。
6. 同一类命中内，置顶优先。
7. 同一类命中内，`updatedAt` 新的优先。
8. 时间相同则 id 更大的优先。

## 8. 验收方式

1. 在首页搜索框输入某个标题关键词，列表应立即只显示匹配便签。
2. 每多输入一个字，结果应实时变化。
3. 删除一个字，结果也应实时恢复。
4. 搜索正文里的词，应能找到对应便签。
5. 搜索 tag 名称，应能找到绑定该 tag 的便签。
6. 先选择“待办”，再搜索关键词，只应在待办范围内搜索。
7. 先选择某个 tag，再搜索关键词，只应在该 tag 范围内搜索。
8. 搜索无结果时，应显示空状态。
9. 处于 tag 筛选状态时点击新建，仍应自动带入当前 tag。

## 9. 已知限制

- 本阶段仍采用本地内存过滤，数据源来自 Room Flow 的当前列表；后续数据量很大时可以把搜索下沉到 Room 查询或 FTS。
- 搜索尚未做高亮显示。
- 搜索框状态暂不跨进程保存。
